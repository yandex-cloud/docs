# Высокая доступность кластера {{ mgp-name }}

Высокая доступность кластера {{ mgp-name }} определяется настройками подключения клиентов, свободным местом в хранилище и другими параметрами.

## Подключение к БД {#connect}

На доступность кластера {{ mgp-name }} влияют используемый клиент и настройки подключения к БД. Чтобы обеспечить доступность кластера:
* Корректно [настройте группы безопасности](../operations/connect/index.md#configuring-security-groups).
* При подключении указывайте оба хоста-мастера (первичный и резервный) либо используйте [особый FQDN первичного мастера](../operations/connect/fqdn.md#fqdn-master).

  {% note warning %}

  Использование особого FQDN первичного мастера приводит к временной недоступности кластера при переключении на резервный мастер.  

  {% endnote %}

* Создайте отдельную [ресурсную группу](../concepts/resource-groups.md) для администрирования кластера, чтобы сохранить возможность подключаться и администрировать кластер даже в моменты, когда лимиты ресурсов в остальных группах исчерпаны.
* Используйте для подключения только [рекомендованные клиенты](../operations/connect/clients.md).

## Свободное место в хранилище {#storage}

Если хранилище баз данных заполнится на 95%, кластер перейдет в режим read-only. Чтобы кластер оставался доступным на запись, регулярно [проверяйте график Disk usage](../operations/monitoring.md#monitoring-hosts) или [создайте алерт](../concepts/storage.md#read-only-monitor).

## Восстановление кластера из резервной копии {#backup}

Если кластер вышел из строя (например, при отказе зоны доступности, в которой находится кластер), его можно восстановить из [резервной копии](../concepts/backup.md).

Чтобы обеспечить сохранность данных и сократить время восстановления кластера:
* Убедитесь, что автоматические резервные копии создаются.
* Перед важными изменениями проводите ручное резервное копирование. Созданные вручную резервные копии являются полными копиями всех баз данных и хранятся бессрочно.
* Проводите плановые тесты восстановления кластера из резервных копий.

## Обслуживание кластера и изменение его параметров {#maintenance}

Запуск [нерегламентных операций технического обслуживания](../concepts/maintenance.md#irregular-ops) может на некоторое время снизить производительность кластера или сделать его недоступным. Установите [время начала обслуживания](../concepts/maintenance.md#maintenance-window) на день недели и час, когда на кластер нет рабочей нагрузки.

[Изменение класса хостов, типа диска или увеличение размера хранилища](../operations/cluster-config.md) прерывает сессии и на некоторое время делает кластер недоступным. Выполняйте эти операции, когда на кластер нет рабочей нагрузки.

{% note info %}

При автоматическом и ручном [резервном копировании](../concepts/backup.md) производительность кластера может снижаться.

{% endnote %}
