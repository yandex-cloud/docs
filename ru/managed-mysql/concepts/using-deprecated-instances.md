# Использование устаревших классов хостов

Начиная с 1 июня 2020 года, классы хостов `s1.nano` и все `m1.*` выведены из эксплуатации для {{ mmy-short-name }}.

Что это значит для пользователей сервиса:
- Теперь вы не сможете изменить класс хостов на `s1.nano` и `m1.*` для вашего кластера.
- Вы сможете продолжать работу с вашим кластером, если в нем используются классы хостов `s1.nano` или `m1.*`, но не сможете [добавлять в кластер новые хосты](../operations/hosts.md) любого класса. 

  Чтобы получить возможность добавлять хосты в кластер, [измените класс хостов](../operations/update.md#change-resource-preset) кластера на один из [действующих классов](instance-types.md).
  
  Например, вы можете мигрировать:
  - с `s1.nano` на класс:
    - `s2.micro` — самый доступный класс хостов со 100% гарантированной долей vCPU. Содержит вдвое больше ресурсов, чем `s1.nano`.
    - `b2.medium` — наиболее близкий по вычислительным ресурсам класс хостов. Стоит дешевле, чем `s1.nano`, но гарантированная доля vCPU ниже: 50% вместо 100%.
  - c `m1.*` на класс `m2.*` (c `m1.medium` на `m2.medium`).

- [Правила тарификации](../pricing.md) для уже созданных хостов `s1.nano` и `m1.*` не меняются.

- [Восстановление кластера из резервной копии](../operations/cluster-backups.md) можно выполнить только на [действующие классы хостов](instance-types.md).

  Вы не сможете восстановить кластер из резервной копии на хосты класса `s1.nano` или `m1.*`, даже если копия была снята с кластера, состоящего из таких хостов.
