В этом руководстве описан порядок действий при утечке [статического ключа доступа](../../iam/concepts/authorization/access-key.md) сервисного аккаунта. Такой ключ может использоваться для доступа к сервисам с AWS-совместимым API, например к [{{ objstorage-full-name }}](../../storage/) или [{{ ydb-full-name }}](../../ydb/). Если ключ попал в публичный репозиторий, артефакт CI, лог, чат или другое место вне контролируемого хранилища, его нужно считать скомпрометированным.

Руководство поможет:

* определить сервисный аккаунт и ресурс ключа по идентификатору `key_id`;
* проверить события об обнаружении и использовании ключа в аудитных логах;
* перевыпустить ключ без остановки рабочих интеграций;
* удалить скомпрометированный ключ и снизить риск повторной утечки.

Чтобы отреагировать на утечку статического ключа доступа:

1. [Подготовьте окружение](#prepare).
1. [Соберите сведения об инциденте](#collect).
1. [Найдите сервисный аккаунт и ресурс ключа](#find-key).
1. [Проверьте события в аудитных логах](#audit-events).
1. [Оцените область возможного воздействия](#scope).
1. [Перевыпустите ключ без простоя](#rotate).
1. [Удалите скомпрометированный ключ](#delete-key).
1. [Проверьте инфраструктуру после отзыва ключа](#post-check).
1. [Снизьте риск повторной утечки](#prevention).

## Подготовьте окружение {#prepare}

{% include [before-you-begin](../../_tutorials/_tutorials_includes/before-you-begin.md) %}

{% include [cli-install](../../_includes/cli-install.md) %}

Установите утилиту [jq](https://stedolan.github.io/jq/download/), чтобы фильтровать JSON-вывод команд CLI.

Для выполнения действий понадобятся роли:

* `iam.serviceAccounts.accessKeyAdmin` или `iam.serviceAccounts.keyAdmin` на сервисный аккаунт, чтобы просматривать, создавать и удалять статические ключи доступа;
* `viewer` или другая роль с правами просмотра на ресурсы, к которым привязан сервисный аккаунт;
* роли для чтения аудитных логов, если вы будете анализировать события через {{ at-name }}, {{ cloud-logging-name }} или {{ yq-name }}.

### Необходимые платные ресурсы {#paid-resources}

Если в вашей инфраструктуре уже настроены аудитные логи, дополнительные ресурсы создавать не нужно. Если аудитные логи еще не собираются, в стоимость поддержки инфраструктуры могут входить:

* плата за хранение аудитных логов в бакете {{ objstorage-name }} (см. [тарифы {{ objstorage-name }}](../../storage/pricing.md));
* плата за хранение записей в лог-группе {{ cloud-logging-name }} (см. [тарифы {{ cloud-logging-name }}](../../logging/pricing.md));
* плата за выполнение запросов к данным, если вы анализируете аудитные логи через {{ yq-name }} (см. [тарифы {{ yq-name }}](../../query/pricing.md)).

{% note info %}

Если {{ at-name }} не был настроен до инцидента, восстановить события за прошлый период с его помощью не получится. Настройте сбор аудитных логов заранее и храните их в месте, доступ к которому ограничен сотрудникам, отвечающим за безопасность и эксплуатацию.

{% endnote %}

## Соберите сведения об инциденте {#collect}

Зафиксируйте исходные данные, прежде чем менять инфраструктуру:

1. Скопируйте уведомление об утечке или ссылку на место, где был найден секрет.
1. Запишите время обнаружения утечки и предполагаемый период, когда ключ был доступен посторонним.
1. Найдите в уведомлении или в найденном фрагменте идентификатор статического ключа доступа `key_id`.
1. Если в уведомлении указан идентификатор сервисного аккаунта, сохраните его. Он понадобится для поиска ресурса ключа.

{% note alert %}

Не публикуйте полный секретный ключ в задачах, чатах, тикетах и PR. Для разбора инцидента достаточно `key_id`, идентификатора сервисного аккаунта, времени обнаружения и ссылки на место утечки, если доступ к нему уже ограничен.

{% endnote %}

## Найдите сервисный аккаунт и ресурс ключа {#find-key}

Если вам известен идентификатор сервисного аккаунта, получите информацию о нем:

```bash
yc iam service-account get <идентификатор_сервисного_аккаунта>
```

Затем найдите ресурс статического ключа. В командах удаления используется идентификатор ресурса ключа `id`, а не публичный идентификатор `key_id`, который используется для аутентификации:

```bash
yc iam access-key list \
  --service-account-id <идентификатор_сервисного_аккаунта> \
  --format json | jq -r '.[] | select(.key_id == "<key_id>")'
```

В результате сохраните значения:

* `id` — идентификатор ресурса статического ключа доступа. Он понадобится для удаления ключа.
* `key_id` — идентификатор ключа, который используется в AWS-совместимых клиентах.
* `service_account_id` — идентификатор сервисного аккаунта.
* `created_at` — дата создания ключа.

Если идентификатор сервисного аккаунта неизвестен, сначала найдите событие обнаружения утечки в аудитных логах. В событии `DetectLeakedCredential` для статического ключа доступа идентификатор сервисного аккаунта находится в поле `details.yandex_cloud_iam_access_key.service_account.service_account_id` или в соответствующем поле JSON-события.

## Проверьте события в аудитных логах {#audit-events}

Проверьте аудитные события, связанные с обнаружением утечки, созданием, использованием и отзывом ключа.

{% list tabs group=at_logs_tools %}

- {{ yq-full-name }} {#yandex-query}

  Выполните запрос к привязке с аудитными логами:

  ```sql
  select
    JSON_VALUE(data, "$.event_time") as event_time,
    JSON_VALUE(data, "$.event_type") as event_type,
    JSON_VALUE(data, "$.details.yandex_cloud_iam_access_key.key_id") as leaked_key_id,
    JSON_VALUE(data, "$.details.yandex_cloud_iam_access_key.service_account.service_account_id") as service_account_id,
    JSON_VALUE(data, "$.details.access_key_id") as access_key_id,
    JSON_VALUE(data, "$.details.last_used_at") as last_used_at,
    JSON_VALUE(data, "$.request_metadata.remote_address") as remote_address
  from bindings.`binding`
  where
    JSON_VALUE(data, "$.event_type") = 'yandex.cloud.audit.iam.DetectLeakedCredential' or
    JSON_VALUE(data, "$.event_type") = 'yandex.cloud.audit.organizationmanager.DetectLeakedCredential' or
    JSON_VALUE(data, "$.event_type") = 'yandex.cloud.audit.iam.AccessKeyLastUsed' or
    JSON_VALUE(data, "$.event_type") = 'yandex.cloud.audit.iam.CreateAccessKey' or
    JSON_VALUE(data, "$.event_type") = 'yandex.cloud.audit.iam.DeleteAccessKey' or
    JSON_VALUE(data, "$.event_type") = 'yandex.cloud.audit.iam.RevokeLeakedCredential'
  order by event_time desc
  limit 100;
  ```

  Чтобы сузить поиск до конкретного ключа, добавьте условие по `key_id` или `access_key_id`.

- {{ cloud-logging-full-name }} {#cloud-logging}

  Используйте фильтр:

  ```sql
  json_payload.event_type = "yandex.cloud.audit.iam.DetectLeakedCredential" or
  json_payload.event_type = "yandex.cloud.audit.organizationmanager.DetectLeakedCredential" or
  json_payload.event_type = "yandex.cloud.audit.iam.AccessKeyLastUsed" or
  json_payload.event_type = "yandex.cloud.audit.iam.CreateAccessKey" or
  json_payload.event_type = "yandex.cloud.audit.iam.DeleteAccessKey" or
  json_payload.event_type = "yandex.cloud.audit.iam.RevokeLeakedCredential"
  ```

{% endlist %}

Сохраните найденные события в тикете инцидента. Если ключ использовался после предполагаемого времени утечки, проверьте журналы сервисов, к которым у сервисного аккаунта был доступ.

## Оцените область возможного воздействия {#scope}

Проверьте, какие действия мог выполнить пользователь с скомпрометированным ключом:

1. Посмотрите роли сервисного аккаунта на организации, облаках, каталогах и отдельных ресурсах. Особое внимание уделите примитивным ролям `admin`, `editor` и ролям с доступом к данным.
1. Проверьте, какие приложения, CI/CD-пайплайны, внешние интеграции и серверы используют этот ключ.
1. Если ключ применялся для {{ objstorage-name }}, проверьте политики доступа к бакетам и журналы операций с объектами.
1. Если ключ применялся для {{ ydb-name }} или другого сервиса с AWS-совместимым API, проверьте журналы и метрики этого сервиса за период возможной компрометации.

{% note tip %}

Если у сервисного аккаунта слишком широкие роли, не ограничивайтесь перевыпуском ключа. После восстановления работы уменьшите набор ролей до минимально необходимого.

{% endnote %}

## Перевыпустите ключ без простоя {#rotate}

Если ключ используется в рабочей интеграции, сначала создайте замену и переключите потребителей, а затем удаляйте старый ключ.

1. Создайте новый статический ключ доступа:

    ```bash
    yc iam access-key create \
      --service-account-id <идентификатор_сервисного_аккаунта> \
      --description "rotated after leaked key <key_id>"
    ```

    Сохраните `key_id` и `secret` нового ключа. Повторно получить значение `secret` будет невозможно.

1. Передайте новый ключ только в защищенное хранилище секретов, например в {{ lockbox-name }}, переменные защищенного CI/CD-окружения или другой утвержденный секрет-хранилище.
1. Обновите приложения и интеграции, которые использовали старый ключ.
1. Проверьте, что новые запросы к сервисам с AWS-совместимым API выполняются успешно.
1. Убедитесь, что старый ключ больше не используется легитимными приложениями.

## Удалите скомпрометированный ключ {#delete-key}

Удалите старый статический ключ доступа по идентификатору ресурса `id`, который вы нашли на шаге [Найдите сервисный аккаунт и ресурс ключа](#find-key):

```bash
yc iam access-key delete <идентификатор_ресурса_ключа>
```

Проверьте, что ключ удален:

```bash
yc iam access-key list \
  --service-account-id <идентификатор_сервисного_аккаунта>
```

Если после удаления ключа рабочая интеграция перестала выполнять запросы, временно верните трафик на исправленную конфигурацию с новым ключом. Не создавайте повторно ключ с теми же учетными данными: значение секретной части статического ключа нельзя восстановить.

## Проверьте инфраструктуру после отзыва ключа {#post-check}

После удаления ключа проверьте:

1. В аудитных логах появилось событие `DeleteAccessKey` или `RevokeLeakedCredential`.
1. В рабочих приложениях нет ошибок аутентификации.
1. За период компрометации не появилось неизвестных ресурсов, новых ключей, политик доступа, бакетов, баз данных или функций.
1. В платежной детализации нет неожиданного роста потребления по ресурсам, к которым мог получить доступ сервисный аккаунт.
1. В месте утечки секрет удален из истории, артефактов и кешей, если такая очистка технически возможна.

Подробнее о действиях с разными типами скомпрометированных секретов см. в инструкции [{#T}](../../iam/operations/compromised-credentials.md).

## Снизьте риск повторной утечки {#prevention}

После завершения инцидента измените способ хранения и выдачи доступа:

* Для {{ TF }} используйте параметр `output_to_lockbox` ресурса `yandex_iam_service_account_static_access_key`, чтобы секретная часть ключа не попадала в файл состояния.
* Для сценариев с {{ objstorage-name }} рассмотрите [эфемерные ключи доступа](../../iam/concepts/authorization/ephemeral-keys.md) или [временные ключи доступа {{ sts-name }}](../../iam/concepts/authorization/sts.md), если они подходят вашему сценарию.
* Для CI/CD в GitHub, {{ GL }} и {{ k8s }} используйте [федерации удостоверений сервисных аккаунтов](../../iam/concepts/workload-identity.md), чтобы не хранить долгоживущие ключи во внешней системе.
* Храните статические ключи в {{ lockbox-name }} или другом защищенном хранилище секретов, а не в репозитории, файлах конфигурации и логах.
* Настройте регулярный поиск событий `CreateAccessKey`, `DetectLeakedCredential`, `AccessKeyLastUsed`, `DeleteAccessKey` и `RevokeLeakedCredential` в аудитных логах.
* Периодически проверяйте роли сервисных аккаунтов и удаляйте неиспользуемые ключи.
