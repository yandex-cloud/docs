# Observability

Метрики и логи в On-premises Yandex Object Storage делятся на две категории:

* Системные метрики и логи, которые необходимы для отладки при инцидентах.
    Их объем сравнительно невелик, но их доступность должна быть высока даже при отказе большинства компонентов Object Storage.

* Используемые в остальных случаях.
    Их объем многократно больше, но допустимо снизить их доступность при инцидентах.

Для хранения метрик и логов применяются Grafana Mimir и Grafana Loki, использующие S3 для долгосрочного хранения данных.

Небольшой объем системных метрик и логов хранится в Minio, все остальные — в служебных бакетах самого Object Storage.

Итоговая схема показана на рисунке:

![Observability](../../_assets/architecture/observability.png =700x)

* Все кластеры Mimir и Loki по умолчанию устанавливаются в трех копиях.
* В этой конфигурации запись и чтение недавних данных доступны при отказе одного хоста. Чтение исторических данных зависит от доступности чтения из соответствующего S3.

Также на схеме отмечены:

* Grafana в stateless-режиме, содержащая набор вшитых дашбордов и доступная при отказе вплоть до двух хостов.
* Эндпоинт формата Prometheus Federation для скрапинга метрик из кластера доступный настолько же, насколько подлежащий Mimir.
* Inspector, являющийся частью Console API и способный собирать системные метрики и логи напрямую из Minio при отказе компонентов Object Storage.
