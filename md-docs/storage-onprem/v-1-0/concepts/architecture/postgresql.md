# Кластеры PostgreSQL

Кластеры PostgreSQL используются для Goose и Zitadel.
По умолчанию все кластеры создаются в схеме 1 primary + 2 replica с подтверждением записи от одной реплики.

Общая схема решения с точки зрения PostgreSQL:

![общая схема решения postgresql](../../_assets/architecture/general-scheme-solution-postgresql.png =700x)

За создание кластеров и управление ими отвечает Cloud-Native PostgreSQL оператор. Потоковую запись WAL и создание резервных копий выполняет Barman. Для хранения резервных копий и WAL в кластер On-premises Yandex Object Storage добавлена инсталляция Minio.

{% note info %}

Инсталляция Minio необходима для повышения надежности PostgreSQL. Например, при восстановлении из резервной копии или синхронизации отстающей реплики извне — в случаях инцидентов или при отказе части компонентов Object Storage.

{% endnote %}

Minio устанавливается с классом избыточности EC 8+8, что позволяет пережить отказ половины (вплоть до восьми) хостов для обеспечения доступности чтения. Как PostgreSQL, так и Minio рекомендуется устанавливать с быстрым SSD/NVME-накопителем для данных.

В такой конфигурации сценарий отказа одного хоста при 1 primary (на этом хосте) + 2 replica выглядит следующим образом:

1. Около 40 секунд (по умолчанию) требуется для обнаружения и подтверждения недоступности primary.
    Запись в кластер PostgreSQL невозможна, чтение возможно с реплик.

1. Как минимум одна из реплик находится в синхронизации с предыдущим primary.
    Выполняется переключение на нее.

1. По окончании переключения доступны чтение и запись.
    Оставшаяся реплика находится в синхронизации с новым primary.

Отказ двух хостов, включая primary и реплику, возможен по двум сценариям:

1. Осталась реплика, которая была в синхронизации с предыдущим primary.
    Тогда сценарий похож на отказ одного хоста, но запись после переключения останется недоступна.

1. Осталась отстающая реплика.
    При переключении на нее она попытается загрузить недостающие WAL из Minio. Здесь возможен сценарий, при котором запись WAL была прервана, и это помешает реплике стать primary без потери данных. Для обеспечения надежности завершение переключения в этом сценарии запрещено. Однако снижается доступность.

Поэтому для обеспечения доступности при отказе двух хостов рекомендуется использовать четыре реплики PostgreSQL с подтверждением записи от двух.

Не следует использовать три реплики и подтверждение от одной из-за возможности split-brain.
