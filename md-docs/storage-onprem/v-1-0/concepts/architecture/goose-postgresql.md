# Работа Goose с PostgreSQL

Все кластеры PostgreSQL ставятся по умолчанию в трех копиях: один primary и две реплики.

Для подтверждения транзакции достаточно подтверждающего ответа (acknowledgement) от одной из двух реплик, в результате чего другая может отставать.

Этого достаточно для баланса отказоустойчивости и производительности.

Так как реплика может отставать, для обеспечения консистентности чтение всегда выполняется с primary.

Fallback на чтение из реплики выполняется только при недоступности primary.

Крайне важно распределение primary серверов S3DB по хостам Kubernetes-кластера — один на хост:

* Primary-шард S3DB создает ощутимую нагрузку на CPU, поэтому при попадании двух шардов на один хост возникнет bottleneck на CPU.
* При отказе хоста запись в находящийся на нем шард S3DB будет временно недоступна.
    * При N хостов (и, соответственно, N шардов S3DB) запись 1/N ключей в S3 становится временно недоступной.
    * Если на хосте располагается более одного шарда S3DB, это сильнее влияет на доступность записи.

Поэтому шарды S3DB располагаются в кластере следующим образом:

![Шарды S3DB в кластере](../../_assets/architecture/s3db-shards.png =700x)

В этой схеме сценарий отказа одного из хостов проходит следующим образом:

1. До выполнения фейловера:
    * недоступна запись 1/N ключей в S3;
    * возникает разовая короткая задержка чтения 1/N ключей из-за fallback на чтение из реплики.

1. После выполнения фейловера чтение и запись доступны.

Отказ двух хостов в данной схеме приведет к невозможности записи 1/N ключей, так как в одном из шардов S3DB останется только один сервер.

Для поддержки записи при отказе двух хостов необходимо аналогично иметь шарды S3DB из пяти серверов с подтверждением записи от двух реплик.
