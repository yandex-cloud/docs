# Elliptics

Elliptics — распределенное хранилище бинарных данных по принципу «ключ-значение», оптимизированное для работы с большим количеством HDD. Позволяет только добавлять новые данные, без изменения или удаления уже записанных (append-only).

На рисунке ниже описаны базовые принципы его работы:

![Принципы работы Elliptics](../../_assets/architecture/elliptics-principles.png =700x)

* На каждый из хостов, содержащих диски для хранилища, ставится один под Elliptics.
* Каждый из подов Elliptics общается со всеми остальными.
* На своем хосте под Elliptics ожидает, что в его пользование эксклюзивно выделены один или несколько дисков.

Пустые диски подготавливаются:

* Elliptics делит каждый HDD-диск на части по 916 ГБ (размер блока отличается для SSD/NVMe);
* такие части называются бэкендами, Elliptics умеет читать и писать в них;
* на уровне выше несколько бэкендов объединяются в капл.

Капл — это ячейка для записи данных в On-premises Yandex Object Storage. Все части бинарных данных всегда либо пишутся в капл, либо читаются из капла.

Размер капла зависит от класса избыточности: например, три бэкенда для репликации x3 или шесть бэкендов для EC 4+2.

На рисунке выше видны четыре капла из трех бэкендов: `[1, 101, 501], [2, 102, 502], [11, 111, 511], [12, 112, 512]`.

Для построения «хорошего» капла необходимо соблюсти условия:

* бэкенды, из которых состоит капл, должны располагаться не менее чем на трех хостах;
* при наличии хостов в разных датацентрах — бэкенды должны быть из разных датацентров;
* даже если несколько частей попадают на один хост (например, при EC 4+2 на трех хостах), они должны быть на разных дисках.

Выполнение этих условий необходимо для:

* надежности хранения данных — отказ диска, хоста или датацентра в геораспределенном варианте приводит к потере только части капла;
* производительности:
    * при загруженности одного из дисков (бэкендов) чтение выполняется из другой части капла;
    * при большом количестве дисков загруженность одного капла по записи не влияет на загруженность большинства других, что позволяет выполнять запись в них;

* доступности данных за счет свойств, описанных выше.

{% note info %}

Серверы Elliptics ответственны только за обработку запросов чтения и записи в бэкенды

За построение «хороших» каплов и расчет весов на запись отвечает [Mastermind](mastermind.md).

За логику работы классов избыточности отвечает клиент Elliptics.

{% endnote %}

На рисунке ниже представлена схема чтения и записи из капла с репликацией x3, которая используется в клиенте Elliptics и обеспечивает баланс надежности и производительности:

![Схема чтения и записи](../../_assets/architecture/elliptics-read-write.png)

* Запись значения выполняется в трех копиях.
* Для подтверждения записи достаточно кворума — двух подтверждающих ответов (acknowledgement) от трех серверов.
* При чтении выполняется параллельный поиск ключа во всех трех бэкендах.
* Чтение будет выполнено из того бэкенда, который ответит быстрее всех.

> **Надежность хранения данных зависит от класса избыточности и количества хостов**
> 
> Предположим, есть три хоста.
> 
> При репликации x3:
> 
> * чтение из Elliptics доступно при отказе не более чем двух хостов;
> * запись доступна при отказе не более чем одного хоста.
> 
> При EC 4+2 чтение и запись доступны при отказе не более чем одного хоста.
> 
> При наличии шести хостов чтение и запись EC 4+2 будут доступны при отказе двух хостов.
> Если хостов пять и меньше, то как минимум две части капла EC 4+2 попадут на один хост.
