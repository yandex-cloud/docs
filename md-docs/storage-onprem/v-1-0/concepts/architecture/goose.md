# Goose

Goose — это сервис, реализующий S3-совместимый API поверх [Elliptics](elliptics.md).

Схема его работы представлена на рисунке ниже:

![Схема работы Goose](../../_assets/architecture/goose-scheme.png =700x)

Каждый под Goose ставится в одной копии на каждый хост Kubernetes-кластера. Это необходимо для эффективной утилизации ресурсов (CPU, RAM) и сети. Этим же обеспечивается доступность и отказоустойчивость самого S3 API.

Для хранения метаданных S3 используются три вида кластеров PostgreSQL:

* S3meta — данные о бакетах, правах и ключах доступа (последние синхронизируются из Cluster API и Zitadel части);
* S3background — служебная база для фоновых задач;
* S3DB — данные об объектах (ключах) в бакетах S3.

{% note info %}

В то время как S3meta и S3background не шардированы, S3DB ставится в количестве шардов, равном количеству хостов.

Этот выбор обусловлен тем, что нагрузка на S3DB многократно выше, чем на остальные БД.

Ключ шардирования — хеш от имени объекта в бакете, что позволяет равномерно распределять нагрузку по шардам.

{% endnote %}

Для чтения и записи Goose также использует информацию из [Mastermind](mastermind.md):

* Пространства имен Mastermind соответствуют классам хранения в S3. Например, пространство имен с каплами с репликацией x3 — класс `STANDARD`, с EC 4+2 — класс `COLD`.
* При выборе капла для записи используются веса, рассчитанные Mastermind.
* Клиент Elliptics, встроенный в Goose, реализует логику работы классов избыточности.

Процесс записи объекта в упрощенном виде:

1. Авторизация, информация о бакете — чтение из S3meta.
1. Выбор шарда S3DB по хешу от имени объекта.
1. Выбор капла для записи согласно классу хранения и весам.
1. Запись в капл Elliptics.
1. Запись в S3DB, включая информацию об ID капла и ключе в Elliptics.

Процесс чтения объекта в упрощенном виде:

1. Авторизация, информация о бакете — чтение из S3meta.
1. Выбор шарда S3DB по хешу от имени объекта.
1. Чтение из S3DB, включая идентификатор капла и ключ в Elliptics.
1. Чтение из Elliptics.

При чтении используется кеширование на уровне Goose после чтения из S3DB для обеспечения консистентности.

Для эффективности кеширования не рекомендуется использовать более одного пода Goose на один хост Kubernetes.
