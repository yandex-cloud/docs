# Типовые проблемы и их устранение

## Задержки VXLAN в Calico {#vxlan-calico-delays}

**Проблема**

Сетевая проблема с VXLAN в Calico, задержки более 63 секунд.

**Решение**

* Вариант 1 — откатите ядро на предыдущую версию.
* Вариант 2 — отключите `checksum offloading`:

    ```bash
    ethtool --offload vxlan-v6.calico rx off tx off
    ```


## Репликация баз данных {#db-replication}

**Проблема**

Отсутствие записей о чанках в `s3-db*` при их наличии в других репликах.

**Решение**

Проверьте синхронизацию между репликами.


## MongoDB {#mongodb}

**Проблема**

Ошибка при запуске из-за недостатка памяти.

**Решение**

Увеличьте выделяемую память.


## Трейлеры и CRC32 в AWS CLI {#aws-cli-trailers-crc32}

**Проблема**

Ошибки при загрузке файлов через свежий AWS CLI.

**Решение**

Добавьте в `~/.aws/config` параметры:

```text
[default]
request_checksum_calculation = WHEN_REQUIRED
response_checksum_validation = WHEN_REQUIRED
```


## Отложенное удаление бакетов {#bucket-delayed-deletion}

**Проблема**

Пользователи сталкиваются с ошибкой `BucketAlreadyOwned` при повторном создании.

**Решение**

Дождитесь фонового удаления бакета.

{% note info %}

Доступны синхронный и асинхронный режимы удаления. По умолчанию установлен асинхронный режим. Чтобы изменить режим, обратитесь в [техническую поддержку](../quickstart.md#support).

{% endnote %}


## Удаление бакетов: GUI показывает 0 объектов {#bucket-deletion-gui-issue}

**Проблема**

GUI показывает 0 объектов, но не позволяет удалить бакет.

**Решение**

Проверьте наличие остатков составных загрузок в бакет.


## Zitadel: Ошибки при инициализации {#zitadel-initialization}

**Проблема**

Ошибки при инициализации.

**Решение**

Четкая процедура переустановки:

1. Удалите пространство имен Zitadel.
1. Удалите PVC и базы данных.
1. Сохраните резервные копии в надежное место.
1. Очистите бакеты в MinIO.
1. Переустановите Zitadel с `initJob: true`.
1. Восстановите базы данных из резервных копий.


## Мониторинг — не отображаются метрики/графики в console UI {#monitoring-metrics-not-displayed}

**Проблема**

Не отображаются метрики или графики в Console UI.

**Решение**

Проверьте место в PVC Prometheus.


## Поды Mastermind (Частые зависания и перезапуски) {#mastermind-pods-hangs}

**Проблема**

Частые зависания и перезапуски подов Mastermind.

**Решение**

Увеличьте таймаут `ready-check` в манифесте.


## Смена пароля Zitadel {#zitadel-default-password-change}

**Проблема**

Необходимость смены пароля по умолчанию в Zitadel.

**Решение**

Используйте nginx для перенаправления на **zitadel-loadbalancer** — с доменом консоли так можно попасть в интерфейс смены пароля.


## WAL и восстановление реплик PostgreSQL {#postgresql-wal-recovery}

**Проблема**

При резком отключении ноды возможна потеря WAL-файлов. Реплики могут не восстановиться из-за отсутствующих WAL-файлов.

**Решение**

Перенесите все WAL-файлы из MinIO, чтобы реплика не искала PITR.


## Восстановление реплик PostgreSQL: нет места на диске {#postgresql-recovery-disk-full}

**Проблема**

Не хватает места на диске.

**Решение**

1. Проверьте свободное место в MinIO:

    ```bash
    kubectl cnpg -n cnpg-system status s3-db01
    ```

1. Воспользуйтесь одним из вариантов:
    * Вариант 1 (предпочтительный): измените размер PVC. Также допустимо для Primary.
    * Вариант 2: удалите PVC.


## Проблемы синхронизации данных в Elliptics {#elliptics-data-sync}

**Проблема**

Бэкенды не синхронизируют данные на диск (метрики `*.need_sync`).

**Решение**

* Вручную запускайте синхронизацию для проблемных бэкендов.
* Используйте мониторинг Grafana.


## Рост TTFB и количества ошибок при увеличении нагрузки {#high-load-ttfb}

**Проблема**

При увеличении нагрузки растет TTFB и количество ошибок (`Reqs: 31027`, `Errs: 1071`).

**Решение**

Добавьте балансировщик (`haproxy`) перед nginx с ограничением 50 конкурентных запросов.


## Установка MongoDB: Ошибка missing featureCompatibilityVersion {#mongodb-featurecompatibilityversion}

**Проблема**

Ошибка `missing featureCompatibilityVersion` при развертывании.

**Решение**

Пересоздайте кластер с удалением PVC.


## Билд каплов в Mastermind {#mastermind-caps-build}

**Проблема**

Проблема с билдом каплов в Mastermind.

**Решение**

1. Проверьте наличие групп через `mastermind group list-uncoupled`
1. Удалите и создайте заново каплы при необходимости
1. Убедитесь, что не осталось старых данных после переустановки.


## Балансировка нагрузки в warp, при round-robin усиливается перекос {#warp-load-balancing}

**Проблема**

При round-robin усиливается перекос.

**Решение**

Используйте `--host-select weighed` в warp.


## Нагрузка на диски {#disk-load}

**Проблема**

На некоторых дисках наблюдается 100% нагрузка.

**Решение**

* Проверьте настройки cpufreq. Возможно, требуется переключение с режима `powersave` на `performance`.
* Равномерно распределите нагрузку между нодами.
* Мониторьте через heatmap или метрики `count(rate() > 0.5)`.


## Возврат дисков после сбоев {#disk-recovery-after-failure}

**Проблема**

Диски не монтируются автоматически после восстановления. Бэкенды остаются незапущенными даже после ручного монтирования.

**Решение**

Воспользуйтесь командой:

```bash
mr_proper --remove-dirs --create-backends -c 0
```


## Восстановление CNPG после падения ноды {#cnpg-recovery-after-node-failure}

**Проблема**

Реплики остаются в состоянии `Waiting for the instances to become active`.

**Решение**

Вручную удалите упавшие поды из состояния `Terminating`.

## Подключения к базе {#database-connection}

**Проблема**

Зависание подключений на 120 секунд при проблемах с сетью.

**Решение**

* Настройте `connection_lifetime` в Goose.
* Добавьте retry на стороне клиента.


## Проблемы с DNS и IPv6 {#dns-ipv6}

**Проблема**

Случайные ошибки `dns server misbehaving` и попытки использовать IPv6.

**Решение**

* Явно укажите IPv4-адреса в конфигах.
* Отключите IPv6 для сервисов.
* Проверьте настройки сети в Kubernetes.

## Смена IP-адресов привела к полному разрушению кластера Kubernetes {#ip-change-kubernetes-destruction}

**Проблема**

Смена IP-адресов привела к полному разрушению кластера Kubernetes.

**Решение**

## Балансировка нагрузки в кластере Kubernetes {#kubernetes-load-balancing}

**Проблема**

Неравномерное распределение нагрузки между нодами.

**Решение**

* Сделайте ручное распределение **1 мастер : 1 нода**.
* Используйте `promote` из [cloudnative-pg](https://cloudnative-pg.io/documentation/1.24/kubectl-plugin/#promote).


## Перекос нагрузки по нодам {#node-load-imbalance}

**Проблема**

* Наблюдается дисбаланс утилизации памяти.
* Различная нагрузка (la).
* Неравномерное распределение IOPS по дискам.

**Решение**

Проверьте и настройте балансировку нагрузки через Load Balancer.


## Производительность при большом количестве потоков и больших объектах {#high-threads-large-objects-performance}

**Проблема**

При нагрузке более 1200 потоков и размере объектов от 5 МБ увеличивается количество ошибок `503`. Кластер не справляется с таким количеством запросов. Особенно критично для схемы хранения **Erasure Coding**.

**Решение**

* Увеличьте таймауты для обработки запросов.
* Используйте **R3** (репликацию в три копии) вместо **Erasure Coding** для критичных нагрузок
* Балансируйте нагрузку между несколькими клиентскими машинами. Например, с трех машин по 400 потоков вместо 1200 с одной.


## Отображение владельца бакета {#bucket-owner-display}

**Проблема**

Невозможно определить владельца бакета через AWS CLI, только через веб-интерфейс. При смене владельца бакета даже создатель теряет к нему доступ.

**Решение**

* В консоли можно увидеть бакет в соответствующем тенанте, если есть права.
* Проверьте настройки на вкладке **Policy**.
