# Логи контейнера

Контейнер пишет логи в Yandex Cloud Logging, в [лог-группу по умолчанию](../../logging/concepts/log-group.md) для каталога, в котором находится контейнер.

Существует два вида логов:
* автоматические — логи запросов к контейнерам.
* пользовательские — логи, которые пишет пользовательский код в стандартный поток вывода (`stdout`) и стандартный поток вывода ошибок (`stderr`). Пользовательские логи имеют минимальный уровень логирования `UNSPECIFIED`.

Подробнее о работе с логами читайте в [документации Yandex Cloud Logging](../../logging/index.md).

## Структурированные логи {#structured-logs}

Кроме текстовых записей, в стандартные потоки `stdout` и `stderr` можно писать структурированные логи в следующем JSON-формате:

* `message/msg` — текст записи.
* `level` — уровень логирования. Доступные уровни логирования — `TRACE`, `DEBUG`, `INFO`, `WARN`, `ERROR` и `FATAL`.
* `stream_name` – имя потока логирования. Длина от 1 до 63 символов.

Все остальные поля JSON автоматически записываются в `json-payload`.

Лог должен быть однострочным. Любая запись, которая содержит поле `message/msg` и длина которой не превышает 64 КБ, считается структурированным логом. Если запись длиннее, она делится на несколько записей и считается текстовой.

Отключить структурированные логи можно, указав переменную окружения `STRUCTURED_LOGGING = false`. Тогда любой лог в JSON-формате будет считаться текстовой записью.

Структурированные логи можно использовать при [одновременных вызовах экземпляра контейнера](container.md#concurrency), чтобы записывать в логи идентификаторы всех вызовов, которые обрабатывал контейнер, а не только последнего из них. Для этого необходимо указать в поле `request_id` уникальный идентификатор вызова. При [режиме работы](container.md#runtime) `HTTP-сервер` он передается в контейнер в заголовке `X-Request-Id`.
