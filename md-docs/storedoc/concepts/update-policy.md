# Политика поддержки версий Yandex StoreDoc

Данный документ описывает жизненный цикл мажорных версий СУБД Yandex StoreDoc.

{% note info %}

СУБД Yandex StoreDoc полностью совместима с [MongoDB](https://ru.wikipedia.org/wiki/MongoDB). Для удобства отслеживания совместимости нумерация версий Yandex StoreDoc полностью соответствует нумерации версий MongoDB (например, версия `Yandex StoreDoc 7.0` полностью совместима с версией `MongoDB 7.0`).

{% endnote %}


## Терминология и этапы жизненного цикла версий Yandex StoreDoc {#version-lifecycle}

Для каждой мажорной версии СУБД в сервисе Yandex StoreDoc действует жизненный цикл продолжительностью **приблизительно 5 лет**, включающий следующие этапы (состояния):

| Состояние версии | Описание и ключевые [действия](#actions-available) | Примерные сроки^1^                     |
|:------|:-------------|:----------------------------|
| Поддерживаемая (`Supported`)  | Версия с действующей поддержкой. **Рекомендуется для всех новых проектов.** | с первого по четвертый год |
| Устаревающая (`Deprecated`) | Версия с завершающейся поддержкой. Создание новых кластеров заблокировано. Существующие кластеры работают в штатном режиме. | пятый год |
| Устаревшая с повышенной стоимостью (`Legacy Extra Paid`)^2^ | Версия, поддержка которой официально прекращена. Кластеры работают, но подлежат тарификации по повышенной ставке. Техническая поддержка ограничена. Сервис создает задачу на принудительное обновление кластеров, которую клиент может отменить через [обращение](https://center.yandex.cloud/support) в службу технической поддержки. | шестой год |
| Выведенная из эксплуатации (`EOL`)^2^ | Версия более не поддерживается. Техническая поддержка не предоставляется. Восстановление из резервных копий не гарантируется. Кластеры работают, но подлежат тарификации по повышенной ставке. | седьмой год и далее |

^1^ Сроки указаны относительно даты релиза мажорной версии. Точные даты перехода между состояниями публикуются в официальных анонсах сервиса.
^2^ Яндекс предварительно [уведомит](#notifications) об изменениях и начале тарификации версий по повышенной ставке в соответствии с условиями Договора.

{% endnote %}


## Ключевые принципы политики поддержки версий Yandex StoreDoc {#key-principles}

Политика поддержки версий Yandex StoreDoc основывается на следующих ключевых принципах:

* **Длительность поддержки**. Каждая мажорная версия находится в состоянии **Поддерживаемой (`Supported`)** в течение приблизительно пяти лет.
* **Поэтапный вывод**. Перед окончанием поддержки версия переходит в состояние **Устаревающая (`Deprecated`)**. После окончания поддержки версия последовательно переходит в состояния **Устаревшая с повышенной стоимостью (`Legacy Extra Paid`)** и **Выведенная из эксплуатации (`EOL`)**.
* **Повышенная стоимость**. При переходе версии в состояние **Устаревшая с повышенной стоимостью (`Legacy Extra Paid`)** к кластерам начинает применяться повышенная ставка тарификации, техническая поддержка ограничивается.
* **Завершение жизненного цикла**. При переходе версии в состояние **Выведенная из эксплуатации (`EOL`)** кластеры продолжают работать, но подлежат тарификации по повышенной ставке, техническая поддержка прекращается, восстановление кластеров из резервных копий не гарантируется.
* **Обновление мажорной версии**. При переходе версии в статус **Устаревшая с повышенной стоимостью (`Legacy Extra Paid`)** сервис создает задачу на принудительное обновление кластера. Клиент может отменить задачу через [обращение](https://center.yandex.cloud/support) в службу технической поддержки.

    {% note alert %}

    Yandex StoreDoc **настоятельно рекомендует выполнить обновление версии** до момента ее перехода в состояние **Выведенная из эксплуатации (`EOL`)**.

    {% endnote %}

* **Уведомление об изменениях**. Сервис заблаговременно [уведомляет](#notifications) клиента об изменениях статуса версии и начале применения повышенной ставки тарификации. Уведомления отправляются за 90 календарных дней до перехода версии в состояния **Устаревшая с повышенной стоимостью (`Legacy Extra Paid`)** и **Выведенная из эксплуатации (`EOL`)**.


## Политика обновлений {#update-policy}

* **Минорные обновления** (в пределах мажорной версии, например, 7.0.12 → 7.0.15) устанавливаются **автоматически** в рамках технического обслуживания, выполняемого в заданное для кластера [окно обслуживания](maintenance.md#maintenance-window) либо по требованию клиента. Эти обновления содержат исправления безопасности и ошибок. Обновление требует кратковременной поочередной перезагрузки хостов кластера.
* **Мажорные обновления** (смена основной версии, например, 7.0.x → 8.0.x) происходят в момент перехода версии из состояния **Устаревающая (`Deprecated`)** в состояние **Устаревшая с повышенной стоимостью (`Legacy Extra Paid`)**. Запланированная операция обновления появляется в интерфейсе Yandex Cloud заранее и может быть отменена через [обращение](https://center.yandex.cloud/support) в службу технической поддержки.

{% note alert %}

Версии в состоянии **Выведенная из эксплуатации (`EOL`)** невозможно обновить до новой мажорной версии. Yandex StoreDoc настоятельно рекомендует выполнить обновление до момента перехода версии в состояние **Выведенная из эксплуатации (`EOL`)**.

{% endnote %}


## Уведомления {#notifications}

Сервис Yandex StoreDoc заблаговременно (за 90 календарных дней) уведомляет клиентов:

* О переходе версии кластера в [состояние](#version-lifecycle) **Устаревающая (`Deprecated`)** (начало действия ограничений). При этом сервис Yandex StoreDoc:

    * уведомляет о невозможности создания новых кластеров с данной версией;
    * рекомендует запланировать обновление существующих кластеров;
    * создает задачу в интерфейсе на принудительное обновление кластеров до новой мажорной версии.

* О переходе версии кластера в [состояние](#version-lifecycle) **Устаревшая с повышенной стоимостью (`Legacy Extra Paid`)** (начало действия повышенной ставки тарификации). При этом сервис Yandex StoreDoc:

    * уведомляет о применении повышенной ставки тарификации;
    * предупреждает о приближающемся сроке окончания поддержки.

Яндекс предварительно уведомит об изменениях и начале действия повышенной ставки тарификации версий в соответствии с условиями договора. Уведомления о таких изменениях отправляются пользователям в соответствии с заданными [настройками уведомлений](../../console/operations/update.md#notifications).
