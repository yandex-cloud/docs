# Обзор сервиса Yandex Cloud Backup


Yandex Cloud Backup — сервис для создания [резервных копий](../../glossary/backup.md) и восстановления ресурсов Yandex Cloud и данных на них. Доступно копирование и восстановление [виртуальных машин Compute Cloud](../../compute/concepts/vm.md), созданных из [поддерживаемых образов Cloud Marketplace](vm-connection.md#os), а также [серверов Yandex BareMetal](../../baremetal/concepts/servers.md) с [поддерживаемыми операционными системами](vm-connection.md#self-install).

{% note info %}

Восстановить резервную копию ВМ на сервер BareMetal или резервную копию сервера BareMetal на ВМ невозможно.

{% endnote %}

Копии ВМ и серверов BareMetal создаются по принципу application-consistent: сохраняются не только данные на дисках, но и данные, которые были отправлены в запись на диск и еще не успели записаться. Такой подход позволяет возобновить работу приложений, запущенных в момент создания копии, сразу после восстановления ВМ или сервера BareMetal из резервной копии. Это важно для ВМ и серверов, которые входят в состав систем хранения данных. Например, когда на ВМ или сервере BareMetal запущена система управления базами данных (СУБД).

Cloud Backup умеет создавать [полные и инкрементальные копии](backup.md#types). В полной копии хранятся все данные ВМ или сервера BareMetal — восстановление происходит быстрее, чем из инкрементальной, но такая копия занимает больше места и создается дольше. В инкрементальной копии содержатся только отличия от предыдущей копии — она создается быстрее и занимает меньше места, но восстановление из такой копии длится дольше, чем из полной. Если вы понимаете, что с момента создания последней копии на ВМ или сервере BareMetal накопилось много изменений, лучше сделать полную копию.

{% note info %}

Если вы используете политику с [опцией](policy.md#specification) быстрого резервного копирования `fastBackupEnabled`, и перед созданием [инкрементальной копии](backup.md#types) выполнялась антивирусная проверка, время резервного копирования может значительно увеличиться. Подробнее в [Взаимодействие с антивирусом](av-interaction.md).

{% endnote %}

Также с помощью Cloud Backup можно восстановить из резервной копии отдельные файлы и директории на любые из ВМ, подключенных к сервису. Подробнее в разделе [Пофайловое восстановление](backup.md#file-by-file). Пофайловое восстановление на серверы BareMetal в настоящий момент невозможно.

Чтобы для ВМ или сервера BareMetal можно было включить резервное копирование, их нужно подключить к сервису и настроить. Подробнее в разделе [Подключение виртуальных машин Compute Cloud и серверов Yandex BareMetal к Cloud Backup](vm-connection.md).

{% note info %}

В момент создания резервной копии виртуальная машина или сервер BareMetal должны быть запущены.

{% endnote %}

После подключения к Cloud Backup и настройки виртуальную машину или сервер BareMetal нужно привязать к [политике резервного копирования](policy.md) — ресурсу Cloud Backup. Политика содержит настройки создания и хранения резервных копий. Вы можете привязывать ВМ и серверы BareMetal к политикам, созданным по умолчанию, или создавать свои политики.

Cloud Backup автоматически поставляет [метрики](../metrics.md) количества защищенных виртуальных машин и серверов BareMetal, а также объема хранилища, занятого резервными копиями, в сервис [Yandex Monitoring](../../monitoring/index.md).

В сервисе действуют ограничения — [квоты и лимиты](limits.md).

## Сравнение со снимками дисков ВМ в Compute Cloud {#snapshot-comparison}

| | Снимки дисков в Compute Cloud | Cloud Backup |
| --- | --- | --- |
| **Гранулярность** | Диск. | ВМ со всеми дисками.<br>Отдельные файлы и папки.<br>Планируется копирование приложений на ВМ. |
| **Типы дисков** | Все типы дисков, кроме локальных на [выделенных хостах](../../compute/concepts/dedicated-host.md).<br>Для [нереплицируемых дисков и SSD-IO](../../compute/concepts/disk.md#nr-disks) время создания снимка не фиксировано. | Все типы дисков. |
| [**LVM**](backup.md#lvm) | Не поддерживается. | Поддерживается. |
| **Восстановление** | Только полное восстановление в новую ВМ. | В новую или существующую ВМ.<br>Частичное восстановление отдельных файлов. |
| **Консистентность** | Неделимый снимок каждого диска. | Неделимый снимок всей ВМ, отдельных папок или файлов. |
| **Проверка целостности резервных копий** | Не поддерживается. | Поддерживается. |
| **Установка агента** | Не требуется. | Требуется, устанавливается автоматически при подключении ВМ к сервису. |
| **Экономичность** | Каждый раз создается полная резервная копия.<br>Сжатие данных не поддерживается. | Создание полных или инкрементальных резервных копий.<br>Несколько уровней сжатия данных с возможностью отключения. |
| **Тарификация** | За каждый 1 ГБ хранимых резервных копий. | За подключение ВМ и за каждый 1 ГБ хранимых резервных копий. |
| **Зависимость от статуса ВМ** | [Снимок](../../compute/concepts/snapshot.md) может быть создан на остановленных или запущенных ВМ. | Снимок может быть создан только на запущенной ВМ с установленным [агентом Cloud Backup](agent.md). |

## Активация сервиса и провайдер резервного копирования {#providers}

Работу Cloud Backup обеспечивает _провайдер резервного копирования_. Сейчас в сервисе доступен один провайдер — Киберпротект.

Для корректной работы [агент Cloud Backup](agent.md), установленный на ВМ или сервер BareMetal, отправляет на серверы провайдера служебные метаданные о конфигурации:

* Имя ВМ или сервера BareMetal.
* Название и версию ОС.
* Версия ядра ОС.
* Внутренний IP-адрес ВМ или сервера BareMetal.
* MAC-адрес каждого сетевого адаптера сервера BareMetal или сетевого интерфейса ВМ.
* Объем оперативной памяти.
* Модель и частота процессора.
* Тип процессора — физический или виртуальный.
* Тип виртуализации — аппаратная или программная [виртуализация](../../glossary/virtualization.md#kak-rabotaet-virtualizaciya) ВМ.
* Имена точек монтирования и [дисков](../../compute/concepts/disk.md), подключенных к ВМ или серверу BareMetal.

{% note warning %}

Данные, хранящиеся на ВМ Compute Cloud или серверах BareMetal пользователя, а также их резервные копии, не покидают пределы Yandex Cloud.

{% endnote %}

Для начала работы сервис нужно активировать — подключиться к провайдеру резервного копирования. Вы можете [активировать](../operations/activate-service.md) сервис напрямую или [подключить](vm-connection.md) к Cloud Backup первую виртуальную машину или сервер BareMetal — в этом случае сервис активируется автоматически.

Чтобы активировать сервис, у вас должна быть _минимальная_ [роль](../security/index.md#backup-user) `backup.user` на [каталог](../../resource-manager/concepts/resources-hierarchy.md#folder), в котором вы хотите создавать резервные копии ВМ или серверов BareMetal.

После активации автоматически создаются следующие политики резервного копирования:
* `Default daily` — ежедневное создание инкрементальной копии с сохранением последних 15 копий.
* `Default weekly` — еженедельное создание инкрементальной копии с сохранением последних 15 копий.
* `Default monthly` — ежемесячное создание инкрементальной копии с сохранением последних 15 копий.
